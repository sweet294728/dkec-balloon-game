# Google Apps Script 排行榜部署說明

這份說明用來建立遊戲的公開前 20 名排行榜。後端會把資料寫入你指定的 Google Sheet；程式碼不包含試算表 ID、正式網址或密鑰，必須由擁有者在自己的 Google 帳號中完成初始化與部署。

## 一、建立與初始化後端

1. 建立一份新的 Google Sheet，建議使用容易辨識的名稱，例如「DKEC 小精靈排行榜」。
2. 在試算表上方選單開啟「擴充功能」→「Apps Script」。
3. 在 Apps Script 編輯器中開啟預設的 `Code.gs`，刪除範例內容，再把本專案 `google-apps-script/Code.gs` 的**完整內容**貼入；不要只貼其中一段，也不要填入任何 Sheet ID、網址或密鑰。按「儲存」。
4. 在編輯器上方的函式選單選擇 `setupLeaderboard`，按「執行」。第一次執行會要求授權；確認目前登入的是這份試算表的擁有者，依 Google 畫面完成授權。
5. 回到試算表並重新整理。確認出現 `Rankings` 工作表、第一列已凍結，而且第一列由左到右正好是以下九個欄位：

   1. `recordId`
   2. `submittedAt`
   3. `nickname`
   4. `normalizedNickname`
   5. `score`
   6. `grade`
   7. `characterId`
   8. `correctHits`
   9. `wrongHits`

如果 `Rankings` 第一列原本已有資料但欄位不符，初始化會停止且不覆寫。請先備份資料，再手動整理第一列為上述九個欄位，然後重新執行 `setupLeaderboard`。

## 二、部署成公開網頁應用程式

1. 在 Apps Script 編輯器右上角選「部署」→「新增部署」。
2. 在類型旁按齒輪，選「網頁應用程式」。可在說明欄填寫版本用途，例如「排行榜正式版」。
3. 「執行身分」選「我」。這會讓程式以部署擁有者的權限存取已初始化的試算表，不會要求玩家登入 Google。
4. 「誰可以存取」選「任何人」或介面中的 `Anyone`／「任何人（包括匿名使用者）」，再按「部署」。
5. 如果 Google Workspace 的組織政策沒有提供匿名的「任何人」選項，請聯絡 Workspace 管理員開放匿名網頁應用程式，或改用允許匿名 Web App 的 Google 帳號部署；不要把玩家登入當作替代方案。
6. 完成部署後，複製正式網頁應用程式網址。正式網址必須以 `/exec` 結尾。`/dev` 是開發測試網址，可能只允許編輯者使用，**不可貼入遊戲**。

## 三、把正式網址接到遊戲

1. 開啟 `src/leaderboard/config.js`。
2. 把剛才複製、以 `/exec` 結尾的正式網址填入 `LEADERBOARD_ENDPOINT` 字串。只放公開部署網址，不要加入 Sheet ID、密鑰或其他憑證。例如保留以下結構並替換空字串：

   ```js
   export const LEADERBOARD_ENDPOINT = '';
   ```

3. 在 `balloon-game` 專案資料夾重新執行：

   ```powershell
   npm.cmd run export:html
   ```

4. 將新產生的單檔 HTML 複製或同步到實際發佈位置。舊的匯出檔不會自動取得剛填入的網址，因此後端部署後一定要重新匯出並替換正式檔。

## 四、後端日後更新

修改 Apps Script 的 `Code.gs` 並儲存後，選「部署」→「管理部署」→目前的正式部署→「編輯」，在版本欄選「新版本」，再按「部署」。使用同一筆部署更新可以保留原本的 `/exec` 網址，遊戲端通常不需要再次更改 `LEADERBOARD_ENDPOINT`；部署後仍應重新跑下方驗收清單。

不要建立新的部署後誤把新、舊網址混用。如果確實改用不同的 `/exec` 網址，必須同步更新 `LEADERBOARD_ENDPOINT`、重新執行 `npm.cmd run export:html`，並替換發佈中的匯出檔。

## 五、隱私與安全限制

- 排行榜只收集玩家主動輸入的暱稱，以及遊戲成績、角色與命中統計；不收集電子郵件、IP 位址、Google 身分或其他個人資料。
- 暱稱和前 20 名成績會透過公開端點提供，視為公開內容。請勿在暱稱輸入真實姓名、聯絡方式或其他敏感資料。
- 這是允許匿名存取的公開端點。後端會驗證欄位範圍、分數與命中關係、在伺服器計算評級，並用鎖避免同時寫入衝突；但公開端點只能提供基本防弊，無法保證完全阻止偽造請求或自動化提交。不要把它描述成完整的防作弊系統，也不要用它保存獎品資格、付款或其他高風險判定。
- 試算表擁有者應定期查看異常內容，必要時先備份再移除不當資料。不要把試算表設成公開可編輯；玩家只需存取部署後的 Web App。

## 六、部署驗收清單

請用正式 `/exec` 網址與實際匯出的遊戲逐項測試：

- [ ] **讀取列表**：尚無資料時，排行榜能正常顯示空狀態；已有資料時，最多顯示 20 筆，順序為高分優先、同分時錯誤命中較少者優先，再依較早提交時間排序。
- [ ] **首次提交**：使用一個新暱稱完成提交，畫面回報成功，`Rankings` 新增一列，`grade` 由後端依分數計算。
- [ ] **更高分取代**：同一暱稱提交更高分後，只保留較佳紀錄，回傳名次與列表都更新。
- [ ] **同分但錯誤較少**：同一暱稱提交相同分數、較少 `wrongHits`，新紀錄應取代舊紀錄。
- [ ] **較差成績保留舊紀錄**：同一暱稱提交較低分，或同分但錯誤較多，後端應回報未更新，原本較佳紀錄與名次保持不變。
- [ ] **大小寫與 NFKC 重複**：先用 `Alice`，再用 `ALICE` 或全形 `Ａｌｉｃｅ` 提交；它們應視為同一個 normalized nickname，不可產生多名玩家紀錄。
- [ ] **公開欄位**：瀏覽器收到的榜單只包含 `rank`、`nickname`、`score`、`grade`、`characterId`，不可出現 `recordId`、`normalizedNickname`、試算表 ID 或錯誤堆疊。
- [ ] **手機／行動裝置顯示**：用實際手機或瀏覽器窄螢幕模式測試結果頁；確認暱稱欄、提交按鈕、狀態訊息和前 20 名列表可閱讀、沒有橫向溢出，並能重新整理榜單。
- [ ] **錯誤處理**：暫時輸入不合規暱稱測試拒絕訊息；玩家端只能看到簡短安全訊息，不可看到授權內容、Sheet ID、原始錯誤或 stack trace。

全部通過後，再將新的單檔 HTML 發佈到正式位置。
